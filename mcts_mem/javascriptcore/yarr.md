- Regular expressions are parsed into a YARR pattern representation and executed by either the YARR interpreter or generated regexp JIT code (`YarrPattern`).
- YARR is the surviving regexp engine; unsupported JIT cases fall back inside YARR rather than into PCRE.
- Regexp JIT availability is controlled separately from the main JavaScript JIT, while still depending on executable assembler support.
- Regex execution is width-aware, carrying the subject's 8-bit or 16-bit representation through parsing, bytecode, and generated code.
- YARR keeps regexp-specific generated code and matching context under VM/RegExp ownership rather than a general PCRE-style library interface.

## Facts

- 2001-08-24 (b33f06b0) rationale: The initial POSIX regexp wrapper operated on ASCII-encoded UString data and did not represent Unicode regexp semantics. (code)
- 2002-12-04 (2b173bf5) statement: POSIX regex was replaced by a vendored PCRE 3.9 copy rather than the OS-provided libpcre; this kept JavaScriptCore independent of the host OS PCRE version and configuration. (uncertain)
- 2004-08-10 (2394c5c0) statement: PCRE_UTF16 compile-time switch and ichar/pcre_char typedefs allow the same PCRE source to build in either 8-bit or 16-bit mode while supporting JSC's internal UTF-16 string representation. (code)
- 2007-11-04 (0fb893f3) measurement: Forking PCRE into a JavaScript-only jsRegExp layer and stripping unused paths produced a reported 0.8% overall and 6.5% regexp SunSpider speedup. (sourced)
- 2010-12-22 (729b6d93) rationale: A dedicated regexAllocator (separate from executableAllocator) was added so that (1) cached regexps share pools with other regexps rather than with large JS function JIT code, reducing pool fragmentation, and (2) a cached regexp cannot hold a large JS-function ExecutablePool alive through the shared-pool refcount. (sourced)
- 2012-09-06 (320753de) pitfall: canUseJIT() and canUseRegExpJIT() both returned m_canUseAssembler directly, ignoring useJIT() and useRegExpJIT() options; when useRegExpJIT() was true the ASM LLInt path inadvertently allowed the full baseline JIT even when useJIT() was false; fix caches separate m_canUseJIT and m_canUseRegExpJIT booleans at construction time. (code)
- 2015-08-13 (795f92bd) rationale: The RegExp code cache exists to help simple loops that recreate the same RegExps, so it is reasonable for periodic code deletion to delete cached RegExp code. (sourced)
- 2017-12-17 (f1f6a09f) rationale: JIT and assembler availability were made static VM queries rather than VM fields because the flags do not need VM ownership and clients should not be bound to a specific VM just to test them. (sourced)
- 2018-02-14 (dff8b0d5) pitfall: The VM-owned RegExp pattern context buffer must be protected by a lock because DFG compilation threads can call into Yarr JIT code concurrently with normal execution. (sourced)
- 2019-04-22 (82568da8) pitfall: VM::canUseRegExpJIT() was a separate cached predicate that depended on canUseAssembler() && Options::useRegExpJIT(), bypassing Options::useJIT(); setting useJIT=false while useRegExpJIT=true would still enable regexp JIT, and enableAssembler() required either useJIT OR useRegExpJIT to be true so disabling both was needed to suppress the assembler. (sourced)

## Moves

- 2002-12-04 (2b173bf5) replaced [[posix-regex-engine]]: POSIX regex (regcomp/regexec) did not support Multiline semantics and had correctness gaps causing multiple real-site form-validation failures; PCRE 3.9 was vendored to fix these. (sourced)
- 2003-06-05 (f03fde27) replaced [[pcre-ascii-passthrough]]: The old code passed UString through .ascii() which truncated all code points above U+00FF to their low byte, corrupting non-ASCII characters in both the pattern and the subject; switching to PCRE_UTF8 mode with explicit UString-to-UTF-8 conversion and bidirectional character/byte offset translation allows correct matching of multibyte characters. (sourced)
- 2004-08-10 (2394c5c0) replaced [[pcre-utf8-round-trip]]: The old mechanism converted UString (UTF-16) to UTF-8 before passing to pcre_compile/pcre_exec, then converted PCRE's byte offsets back to UTF-16 offsets after matching; eliminating the round-trip conversion by extending PCRE to operate natively on uint16_t removes the allocation and offset-translation overhead on every regex match. (sourced)
- 2005-09-09 (9b1f5040) replaced [[pcre-monolithic-5x]]: PCRE vendored library updated from ~5.x single-file layout to PCRE 6.1 modular multi-file layout including Unicode character property support (ucptable.c, ucp_findchar.c); the upgrade preserves Apple's UTF-16 patch applied on pcre-6-1-branch, gaining new PCRE 6.1 features and bug fixes that the old monolithic pcre.c could not provide. (sourced)
- 2007-02-06 (49f7c83a) replaced [[pcre-no-recurse-setjmp-longjmp]]: setjmp/longjmp in the PCRE NO_RECURSE heap-frame path caused a 25-30x regexp slowdown on macOS 10.5 vs 10.4; Shark profiling identified setjmp overhead as the root cause; GCC computed-goto (&&label, goto *ptr) eliminates the setjmp save/restore entirely. (sourced)
- 2007-07-19 (2cd3a93e) replaced [[regexp-sanitize-pattern-workaround]]: sanitizePattern pre-processed \uXXXX escapes at the KJS level by converting them to UTF-16 characters and inserting backslashes before special regex metacharacters, but this was fragile (the switch of characters to escape was hand-enumerated) and did not fix the underlying PCRE length-preflighting bugs caused by unsupported advanced features; adding a JAVASCRIPT-mode escape table inside PCRE natively handles \u and \v and also allows systematically disabling all non-JS features (named recursion, POSIX classes, etc.) that caused incorrect preflight lengths. (sourced)
- 2007-11-04 (0fb893f3) replaced [[pcre-public-api]]: The standard PCRE public API (pcre_compile2/pcre_exec/pcre prefix naming) included unused features (UTF-8, POSIX, UCP, study, etc.) that cost code size and execution overhead; renaming to jsRegExpCompile/jsRegExpExecute and stripping all JS-unused paths gave 0.8% overall / 6.5% regexp SunSpider speedup while intentionally forking from upstream PCRE. (sourced)
- 2007-11-04 (f26e598e) replaced [[pcre-generic-single-char-opcodes]]: Generic single-character match opcodes applied full Unicode logic even for pure ASCII patterns; adding OP_ASCII_CHAR and OP_ASCII_LETTER_NC specialized opcodes skipped surrogate-pair checks and used efficient ASCII case comparison, yielding 2.6% overall / 32.5% regexp SunSpider speedup. (sourced)
- 2007-11-07 (7f105b01) dropped: POSIX regex backend: POSIX regex backend removed with no successor because author could not test it after refactoring RegExp to be ref-counted and the existing PCRE (jsRegExp) backend was the primary path. (sourced)
- 2007-11-11 (74afd231) replaced [[pcre-c-api-with-jsregexpchar]]: The PCRE C-language layer used a custom JSRegExpChar (unsigned short) type and a combined int options bitfield; converting to C++ with UChar and typed enum parameters eliminated the type mismatch with WebKit's UTF-16 strings and enabled C++ new/delete, removing the pcre_malloc/free indirection and FastMallocPCRE.cpp shim. (sourced)
- 2007-11-29 (09bb02e6) replaced [[pcre-recursion-check-in-hot-loop]]: Moving the recursion-depth check from the top of the RECURSE label (executed on every frame re-entry including returns) to inside pushNewFrame (executed only when a new frame is allocated) removes the check from the super-hot steady-state path, contributing to the 8% speedup. (sourced)
- 2008-01-01 (81c47fd1) replaced [[pcre-bracket-empty-match-detection-removed]]: A prior optimization commit removed the eptrblock stack that tracked bracket-start positions to detect infinite loops on brackets matching the empty string; this caused jsRegExpExecute to return -2 on 34 test cases; the stack must be maintained to prevent infinite repetition of zero-length matches. (code)
- 2008-03-31 (2fb55858) replaced [[pcre-circ-doll-unified-opcodes]]: The unified CIRC/DOLL opcodes checked the multiline flag at match time on every execution, preventing the anchored-regex optimization in jsRegExpExecute (which requires knowing at compile time whether the pattern is anchored); splitting into separate compile-time opcodes (CIRC=single-mode, BOL=multiline) allows branchNeedsLineStart to detect anchoring statically and skip the retry loop for anchored patterns. (code)
- 2008-12-05 (3fde03e7) replaced [[wrec-parentheses-pcre-fallback]]: The old parseParentheses immediately returned false with UnsupportedParentheses (forcing PCRE fallback) for all parenthesized subexpressions; the new code adds native JIT codegen for (?=) and (?!) lookahead/lookbehind assertions while still falling back to PCRE for capturing and non-capturing groups. (code)
- 2009-02-10 (a4e24f08) replaced [[wrec-x86-64-regparm-arg-shuffle]]: When WREC was ported to x86-64 it reused x86 register assignments and emitted argument-shuffle moves (rdi->eax, rsi->edx, rdx->ecx, rcx->edi) in generateEnter because x86-64 SysV ABI passes args in rdi/rsi/rdx/rcx, not via regparm(3); switching to native ABI register names eliminates these shuffles and aligns the register allocation with the calling convention. (code)
- 2009-04-24 (0f668258) replaced [[wrec-regex-jit]]: YARR JIT (Yet Another Regex Runtime) was enabled by default on x86/x86-64 Mac and Windows, replacing WREC, after YARR reached sufficient correctness to pass tests on those platforms. (sourced)
- 2009-04-28 (463dacc4) replaced [[wrec-regex-engine]]: WREC (WebKit Regular Expression Compiler) replaced by YARR (Yet Another Regex Runtime) on Mac and Windows x86/x86-64; YARR provides both a JIT tier and a bytecode interpreter fallback, while WREC had only a JIT path. (code)
- 2010-02-27 (7faa2fd9) removed: All builds had switched to YARR, making WREC dead code; the commit message states 'All builds should have switched to yarr by now.' (sourced)
- 2010-11-21 (d77d2f20) replaced [[pcre-yarr-tri-mode-regex]]: ENABLE_YARR macro and PCRE fallback path were removed; YARR JIT now falls back to the YARR interpreter (not PCRE) when JIT compilation is unsupported, eliminating the PCRE dependency from JSC. (sourced)
- 2011-02-10 (5f1c8764) removed: PCRE source removed from tree because Yarr had already been adopted as the JSC regex engine, leaving PCRE dead code that was still being built; sourced from bug 54188 and 'Remove PCRE source from trunk' message. (sourced)
- 2010-05-09 (48eeb64a) replaced [[yarr-per-platform-enable-rules]]: Per-port YARR enable rules (enumerating specific CPU/OS/platform combos) were replaced with a single rule enabling YARR and YARR_JIT whenever JIT is enabled, since YARR requires JIT and the redundant guards were scattered per-port. (code)
- 2012-09-05 (eed2cdb0) replaced [[yarr-jit-controlled-by-usejit-flag]]: YarrJIT uses the assembler for regex compilation only and does not require the full JIT infrastructure; tying it to useJIT() prevented enabling regex JIT on the LLInt C loop backend (and similar configurations) where the main bytecode JIT must be disabled but the assembler is still accessible, so a separate useYarrJIT() option was introduced to allow independent control. (sourced)
- 2018-02-14 (dff8b0d5) replaced [[yarr-stack-pattern-context-buffer]]: A stack-local Yarr parenthesis context buffer could overflow the stack, so the buffer moved to a lazily allocated VM-owned buffer acquired only for JIT code that declares it uses pattern context. (code)
