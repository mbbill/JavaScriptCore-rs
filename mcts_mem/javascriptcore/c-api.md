- The public C API exposes opaque pointer handles for contexts, values, objects, strings, classes, property-name arrays, and context groups; conversions to JSC internals are centralized in the API cast layer. (`APICast.h`)
- `JSGlobalContextRef` is the owning context handle, while `JSContextRef` is a non-owning view of a global object and `JSContextGroupRef` names the VM/object-space shared by related contexts.
- `JSValueRef` remains pointer-typed at the ABI boundary: 64-bit builds bit-cast the encoded JSValue word, while 32-bit builds box non-cell values in API wrapper cells to preserve a pointer representation. (`JSAPIValueWrapper`)
- Public strings are context-free ref-counted `OpaqueJSString` objects whose storage is isolated from identifier tables, preserves 8-bit strings, and lazily materializes a 16-bit view only for legacy character-pointer APIs.
- Embedder object behavior is described by a versioned `JSClassDefinition` and stored in ref-counted `OpaqueJSClass` objects; per-global class state holds static callback tables and cached prototypes separately from the context-free class descriptor.
- C callback objects carry a retained class descriptor, raw private data, and GC-visible private-property slots; property, call, construct, conversion, and finalization callbacks run through API shims that translate between opaque handles and VM values.
- Exception flow across C API evaluation, conversion, call, construct, and callback entry points is explicit through nullable return values and `JSValueRef*` exception out-parameters rather than a context-global exception slot.

## Facts

- 2006-06-21 (bfb69497) rationale: API types are opaque forward-declared C structs cast to internal C++ pointers, keeping the ABI pure C and hiding the internal class hierarchy from embedders. (code)
- 2006-07-04 (3d3b123c) rationale: evaluation switched from bool success plus blended return/exception out-parameter to direct `JSValueRef` return with an explicit exception out-parameter, matching C API conventions and making call-site exception handling explicit. (sourced)
- 2008-07-23 (26073881) pitfall: `JSClassRef` is created without a context and must be usable across contexts, so per-context identifier/static-table/prototype state cannot live in `OpaqueJSClass`; it lives in `OpaqueJSClassContextData` keyed by the global data/context. (sourced)
- 2010-01-26 (6fe32003) pitfall: API entry shims, not callback shims, own timeout-checker start/stop because the checker monitors VM execution rather than client callback execution. (sourced)
- 2013-03-22 (c1225a67) rationale: per-class context data moved to `JSGlobalObjectRareData` so a `JSClassRef` prototype cache is scoped to the owning context and unused globals pay no HashMap cost. (sourced)
- 2014-01-17 (ca8bd025) rationale: `OpaqueJSString::characters()` lazily allocates a 16-bit buffer for the legacy `JSStringGetCharactersPtr` view while preserving 8-bit backing storage when possible. (code)
- 2017-05-11 (84a0d815) rationale: `JSWeakCreate` / retain / release take `JSContextGroupRef` because they need only VM access and do not execute arbitrary JavaScript. (sourced)
- 2020-04-19 (2ea571e8) pitfall: public API entry points must return a benign sentinel immediately after propagating an exception because the nominal result may be garbage and `toRef()` is not guaranteed to sanitize it on every architecture. (sourced)
- 2026-04-14 (3572482f) statement: the GC dangling-reference crash path points API clients at `JSValueUnprotect` misuse, unprotected heap `JSValueRef` storage, destroyed contexts, and cross-context-group use as boundary-lifetime failures. (code)

## Moves

- 2006-07-02 (fe2e681b) replaced [[js-c-api-per-object-callbacks-struct]]: JSObjectCallbacks inline struct embedded per-object (stored as m_callbacks copy) could not express static property/function tables shared across objects, forced objects onto the oversized heap, and could not support class-level inheritance chains; JSClassRef is a ref-counted shared class object with HashMap<UString,StaticValueEntry> and HashMap<UString,StaticFunctionEntry> tables, removing the need for a getProperty callback for most lookup cases. Specialized subclasses JSCallbackFunction and JSCallbackConstructor preserve JSFunctionMake/JSConstructorMake without creating a custom class per call. (sourced)
- 2006-07-10 (be86fd6e) replaced [[jsc-api-opaque-types-void-ptr]]: JSValueRef changed from void* to const struct __JSValue* and JSObjectRef from struct __JSObject* to struct __JSValue* to gain C type-safety: the compiler now rejects passing a JSObjectRef where a JSValueRef is expected and vice versa, and revealed numerous existing bugs in the API implementation. (sourced)
- 2006-07-11 (c0949c09) replaced [[jsc-api-exception-context-functions]]: Per-context exception get/set/clear functions were replaced by JSValue** exception out-parameters on every callback signature, using exec->exceptionSlot() to provide the slot; this makes exception flow explicit at each call site rather than requiring callers to poll/set a context-global exception state. (sourced)
- 2006-07-14 (723987b9) replaced [[jsc-api-context-destroy-ownership]]: JSContextDestroy (manual delete) was replaced by retain/release ref-counting because embedding environments need shared ownership of contexts; also splits JSContextRef (non-owning, const) from JSGlobalContextRef (owning) to enforce correct usage at the type level. (sourced)
- 2006-07-16 (c81ff31a) replaced [[jsobjectcallbacks-flat-args]]: Packing all class-creation parameters into a single versioned struct (JSClassDefinition) enables ABI-stable forward migration and allows adding new fields (e.g., className) without breaking callers. (sourced)
- 2006-07-16 (6321480d) dropped: callback-object private data on callback functions and constructors — For callback functions and constructors the native implementation IS the stored callback, so also supporting private data would confuse the idiom that private data = native implementation; removed to keep the ownership model unambiguous. (sourced)
- 2006-10-12 (622e7bb8) dropped: JSObjectMakeWithPrototype API — The authors concluded the API encouraged an anti-pattern (manual prototype management) that goes against JavaScript practice of shared prototypes; removing JSObjectMakeWithPrototype and renaming kJSClassAttributeNoPrototype to kJSClassAttributeNoAutomaticPrototype forces callers to use JSObjectSetPrototype explicitly if they need manual prototype control, reducing surface area for misuse. (sourced)
- 2008-08-15 (f7218e4f) replaced [[jsstring-ref-as-ustring-rep-cast]]: JSStringRef was a raw reinterpret_cast alias for UString::Rep*, which caused the public API string to be silently linked into an internal identifier table, breaking the implicit API contract that strings are context-free. (sourced)
- 2010-01-22 (56810cc5) replaced [[opaque-jsclass-conditional-prototype]]: Always creating a prototype class for every OpaqueJSClass (not just when staticFunctions or parentClass is present) ensures prototype chains are always correctly hooked up and instanceof works on all API classes, not just those with static functions. (sourced)
- 2012-10-03 (40a1198a) replaced [[opaque-jsstring-raw-ucharbuffer]]: OpaqueJSString stored a manually heap-allocated UChar* buffer which always forced 16-bit storage even for 8-bit strings; replacing it with a WTF::String member preserves 8-bit string encoding and eliminates manual memory management (new[]/delete[]). (sourced)
