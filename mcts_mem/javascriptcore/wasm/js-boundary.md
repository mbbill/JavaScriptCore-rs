- JS-to-wasm, wasm-to-JS, wasm-to-wasm imports, and JS-to-wasm inline caches use distinct callees, compilation modes, and wrapper/exit-stub code paths. (`JSToWasmCallee`)
- `JSWebAssemblyInstance` keeps memory caches, import call-link info, tables, globals, baseline data, GC structures, and allocators in one trailing-array allocation.
- Imported function descriptors carry target instance, entrypoint, boxed callee, import stub/function, and optional call-link info for one callable shape across direct imports, tables, references, and jitless exits.
- Function tables store wasm-side metadata and materialize `WebAssemblyFunction` wrappers lazily when JavaScript observes a slot.
- Boundary marshalling is generated from the wasm calling convention, including boxed JS returns, i64/BigInt decisions, V128 rejection paths, and multi-result return unpacking.
- WasmGC reference checks and struct/array accesses use RTT or StructureID identity and logical field/element heaps rather than JS object-shape assumptions.

## Facts

- 2017-04-04 (a36e2434) rationale: the Wasm-to-JS stub still stores a JSCell callee because the call IC infrastructure that writes the inline cache expects a JSCell callee, so structured cloning of Wasm modules must regenerate those JS call stubs. (sourced)
- 2018-10-01 (d2545449) statement: JSToWasm glue does not have tag registers loaded; boxInt32 and boxDouble use DoNotHaveTagRegisters mode, and boxDouble gained that mode in this change. (sourced)
- 2023-01-23 (3563dd18) measurement: the direct FTL JS-to-Wasm path improved JetStream2/richards-wasm Runtime from 13.021 to 16.129, a 23% improvement. (sourced)
- 2023-01-25 (0ebf892b) rationale: fixed-size FuncRefTable exists because many funcref tables have initial size equal to maximum size, and that invariant lets BBQ and OMG avoid one load indirection for non-import table calls. (sourced)
- 2025-01-17 (717c7964) pitfall: when a WasmToJS DataOnlyCallLinkInfo is materialized after instance construction, it must be fully initialized before publication and followed by a JSWebAssemblyInstance write barrier because concurrent GC may already visit the slot. (code)
- 2025-02-01 (a6731279) rationale: Wasm signatures cache whether any argument or result is i64, so JS-to-wasm and wasm-to-JS entrypoint selection can avoid rescanning every parameter and result during startup/compilation. (code)
- 2025-08-29 (0237d6e3) pitfall: on 32-bit JS-to-Wasm frame construction, storing the boxed native callee pointer without also storing NativeCalleeTag leaves a stale tag that stack walking may interpret as a JS object. (code)
- 2025-10-22 (2db05791) rationale: the JIT uses a shared polymorphic-callee thunk so compiled indirect calls can update per-callee counters in fast code and materialize the PolymorphicCallee side table only on first polymorphic observation. (code)
- 2026-01-22 (ea11204f) statement: ref.cast as a B3 value lets ReduceStrength remove trapping bits from following WasmStructGet/Set when the cast proves non-null, and convert a nearby allow-null cast plus trapping field access into a non-null trapping cast that can use the signal-handler null trap. (code)
- 2026-03-04 (73db6545) pitfall: Wasm-to-JS argument marshalling operations that can allocate BigInts need a WasmOperationPrologueCallFrameTracer because allocation can update ShadowChicken while the synthetic call frame is live. (sourced)

## Moves

- 2017-06-08 (6d0ffd96) replaced [[eager-paired-wasm-entrypoints]]: Only functions reachable from exports, element segments, or the start function need JavaScript-to-WebAssembly wrappers; internal-only functions keep only their wasm entrypoint. (code)
- 2018-10-01 (d2545449) replaced [[wasm-call-runtime-return-type-switch]]: The return-type switch at call site interpreted the raw result at runtime using a runtime switch over signature.returnType(); since returnType() is known at compile time, the conversion can be emitted as JIT code into the JSToWasm glue, eliminating the runtime dispatch and the need for tag registers at the call site. (code)
- 2018-11-19 (7522c43a) replaced [[wasm-i64-check-at-call-site]]: Moving I64 argument/return type check from the runtime call-site stub to the compiled JSToWasm wrapper encodes the check once at compile time rather than on every invocation, and is a prerequisite to removing callWebAssemblyFunction entirely. (code)
- 2024-12-18 (602054ed) replaced [[wasm-import-function-info-instance-tail]]: The jitless wasm-to-JS thunks need one descriptor that carries type index, import JS function, call-link info, target instance, entrypoint, and boxed callee data for both direct imports and table/ref calls, while the old instance-tail ImportFunctionInfo and wasm-only table entries could not represent JS call targets from tables or pass the descriptor through the non-JIT frame. (code)
- 2025-01-17 (717c7964) replaced [[embedded-wasm-import-call-link-info]]: The import record stopped embedding DataOnlyCallLinkInfo because not all imports are JS calls and JIT-less calls through FuncRefTable/WebAssemblyFunctionBase need a maintained CallLinkInfo pointer to the original instance slot. (code)
- 2025-10-24 (2c1627d7) replaced [[split-wasm-to-js-exit-result-unpacking]]: The old IPInt exit path used separate operations to decide whether to unpack multi-value JS results and then iterate them, while the new return marshalling operation handles zero, one, and multi-result cases itself and writes results according to the wasm callee convention. (code)
- 2026-01-16 (4758c065) replaced [[wasmgc-struct-tbaa-by-field-type-and-index]]: Struct-field alias keys now use the RTT that defines an inherited field plus the field index because unrelated WasmGC struct types can share the same field offset and value type while inherited A.a/B.a/C.a must still alias. (code)
- 2026-01-22 (ea11204f) replaced [[omg-inline-wasm-ref-type-check-lowering]]: Wasm ref.cast/ref.test became explicit B3 values so B3 data-flow analysis, CSE, and ReduceStrength can reason about WasmGC type checks before they are lowered to branches, checks, patchpoints, and loads. (code)
- 2026-03-27 (e40eb8ad) replaced [[wasm-gc-object-embedded-rtt-type-checks]]: After realm-less interning made Wasm GC Structure identity unique per RTT, type checks could use object StructureID and an inlined StructureID display instead of an RTT pointer stored in every object. (code)
- 2026-05-17 (ca0c6bb1) replaced [[eager-wasm-table-function-wrappers]]: Active element segments now install wasm-side function metadata and materialize WebAssemblyFunction wrappers only when JS observes the slot, avoiding wrapper/JSToWasmCallee allocation for entries used only by wasm call_indirect. (code)
