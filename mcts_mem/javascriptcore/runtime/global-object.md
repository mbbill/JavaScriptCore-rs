- Each realm is rooted at a `JSGlobalObject` that owns or lazily caches the constructors, structures, global properties, and per-realm runtime state used by code executing in that realm.
- The global object is the rendezvous point for realm-specific watchpoints and caches: lexical shadowing, species/prototype-chain sanity, structure-cache invalidation, and pending asynchronous work all attach to the realm rather than to scattered prototype objects.
- Expensive or rarely used realm state is materialized lazily through global-object properties instead of being allocated during global initialization.
- Cross-realm language behavior is represented by the callee or lexical global object, not by process-global interpreter state.

## Facts

- 2007-10-25 (7efe5eb6) measurement: replacing the static interpreter lookup map with a typed `JSGlobalObject` back-pointer removed map maintenance and produced a 0.5% SunSpider speedup. (sourced)
- 2010-08-24 (e37f9518) pitfall: VM-global, time-seeded `Math.random()` leaked identical predictable sequences across global objects; per-global-object random state with a cryptographic seed removed both predictability and cross-realm sharing. (sourced)
- 2019-01-21 (bbdc7b7b) rationale: module-loader state moved to `LazyProperty` because contexts that never use ES modules should not pay allocation and structure-creation cost during `JSGlobalObject::init`. (sourced)
- 2019-03-15 (72839588) rationale: WebAssembly and iterator-result structures became lazy global properties because eager initialization increased startup memory for environments that never use those features. (sourced)
- 2019-03-13 (3b42c3ac) rationale: `Array.prototype[Symbol.species]` watchpoint ownership moved to the global object to avoid an ArrayPrototype finalizer and WeakBlock cost and to centralize lazy watchpoint installation. (sourced)
- 2022-10-10 (b8479830) rationale: prototype-chain sanity checks became JSGlobalObject-owned absence-of-indexed-properties watchpoints so DFG and FTL can guard one reusable global condition instead of scattering structure-transition watchpoints. (code)
- 2022-04-13 (ced57272) rationale: cached structures are tied to a realm, so the structure cache moved from a VM-wide map keyed by global object to a JSGlobalObject-owned cache. (code)
- 2024-06-08 (1e4577ba) pitfall: strong roots in `DeferredWorkTimer` could keep unresolved `Atomics.waitAsync` promises and their global objects alive forever; moving ownership to the target global lets GC collect the cycle and cancel waiters when the global dies. (sourced)

## Moves

- 2007-10-25 (7efe5eb6) replaced [[interpreter-map]]: A static HashMap<JSObject*, Interpreter*> was needed to look up the Interpreter from a global object; introducing JSGlobalObject with an embedded m_interpreter back-pointer eliminates the hash-map lookup and map maintenance at init/destroy time, giving a 0.5% SunSpider speedup and removing the static data structure. (sourced)
- 2007-12-01 (8104e1bb) replaced [[interpreter-refcount-ownership]]: Circular mark-graph and ref-count complexity arose because Interpreter marked JSGlobalObject and was itself ref-counted by external holders; reversing ownership so JSGlobalObject (a GC object) directly owns Interpreter (via std::auto_ptr) lets GC protect both through a single gcProtect(globalObject) call, eliminating special GC treatment for Interpreters. (sourced)
- 2008-04-26 (9e909d16) removed: cached-page partial save/restore — CachedPage now stores the full live JSGlobalObject rather than a partial snapshot; the snapshot approach (SavedBuiltins + SavedProperties) required manually listing every built-in constructor/prototype to save and was brittle as the object grew. (sourced)
- 2019-01-21 (b145cc48) replaced [[scope-cache-invalidation-per-binding-watchpoint]]: The old mechanism iterated all live CodeBlocks and all op_resolve_scope instructions to clear per-entry caches whenever any lexical binding was added; the new epoch counter stored in op_resolve_scope metadata and compared inline in LLInt/JIT avoids the O(codeblocks × instructions) scan and also handles the case where a global property was deleted then re-shadowed without requiring property-specific watchpoints. (code)
- 2020-04-09 (0ec50c9b) replaced [[module-environment-global-object-parent]]: Making JSGlobalLexicalEnvironment the module environment's upper scope lets module bytecode use the same global-property watching and exit-site tallying as scripts. (code)
- 2022-10-12 (4d0bfb7d) replaced [[string-prototype-sane-chain-structure-watchpoints]]: StringPrototype sane-chain checks moved to a JSGlobalObject watchpoint set so DFG/FTL no longer scatter ad hoc transition watchpoints and uDFG can query one global-object condition. (sourced)
- 2022-05-22 (fd038f44) replaced [[having-bad-time-watchpoint-guards-structure-cache]]: A StructureCache can contain Structures whose prototype chains involve globals going bad even when the cache-owning JSGlobalObject is not itself having a bad time, so DFG must guard cached object structures with a cache-clear watchpoint rather than only the global's having-a-bad-time watchpoint. (code)
- 2024-06-19 (2aa59a0d) replaced [[globalobject-ticket-barrier-map]]: WeakHashSet lets JSGlobalObject track DeferredWorkTimer tickets through weak ownership instead of TicketData destructors manually unregistering a HashMap of write barriers. (code)
