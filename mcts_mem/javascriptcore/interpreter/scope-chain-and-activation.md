- Lexical lookup is represented by explicit scope/environment objects linked through the scope chain; captured state is materialized only when execution can observe it after the stack frame would otherwise disappear.
- Variable storage distinguishes stack-frame slots, activation/lexical-environment storage, global object properties, and global lexical symbol-table bindings.
- Parser and bytecode metadata decide which variables require heap storage, TDZ tracking, dynamic lookup, or global declaration side effects.
- Scope slow paths use shared JSScope and symbol-table mechanisms instead of duplicating lookup logic in interpreter, LLInt, and JIT helpers.

## Facts

- 2002-11-22 (f2cbe708) measurement: inlining ActivationImp in ContextImp and stopping GC collection of separate activation objects gave 3% on iBench. (sourced)
- 2002-11-22 (2d8935af) measurement: changing ScopeChain from doubly-linked copy-on-push to singly-linked shared tails gave 11% on iBench. (sourced)
- 2008-01-01 (8b39be79) measurement: embedding the top ScopeChain node in the ScopeChain object and allocating only on sharing yielded a 1.019x SunSpider speedup before that form was reverted. (sourced)
- 2008-09-29 (d59ec5ce) rationale: isActivationObject only affected assertions and exception-stack unwinding, but StructureID checks are the canonical JSC identity mechanism and removed a redundant vtable slot. (sourced)
- 2011-03-04 (0094382f) rationale: the inline-field revert kept the WriteBarrier<Unknown> register representation because it was a separate correctness fix from the failed layout change. (sourced)
- 2011-03-10 (6ca5afe1) rationale: inlining JSVariableObject fields removed a d-pointer load from generated variable access, though no commit message recorded the full design reason. (uncertain)
- 2011-05-18 (f26e9bca) pitfall: JSGlobalObject must be constructed with a pre-created Structure because allocating one inside the constructor can trigger GC before the object is fully initialized. (sourced)
- 2012-01-13 (2a60bfd6) rationale: JSActivation needs dynamic scope checks for eval-using functions only outside strict mode because strict eval cannot introduce variables. (sourced)
- 2012-08-28 (ef24353d) rationale: JSScope centralizes resolve/resolveSkip/resolveGlobal/resolveBase logic that had been duplicated in Interpreter.cpp, JIT stubs, and slow paths. (sourced)
- 2013-05-09 (4e61ee97) pitfall: a single global eval CodeCache could reuse unlinked code with scope offsets from a different function or with/catch context; eval caching is now scoped to safe owner code blocks. (code)
- 2019-03-21 (640a6116) rationale: JSGlobalObject and JSGlobalLexicalEnvironment are allocated rarely enough that per-object finalizers replace a dedicated segmented-variable-object subspace with no measurable cost. (sourced)
- 2026-03-29 (59d83df3) pitfall: a global-scope lexical variable that is resolved dynamically still needs lexical watchpoint invalidation; treating non-captured globals as ordinary globals can miss later lexical shadowing. (code)

## Moves

- 2002-11-22 (f2cbe708) replaced [[scope-chain-as-list]]: List was used for scope chains but carries ref-counting and mark() overhead tuned for argument lists; separating ScopeChain allows independent optimization and removes needsMarking complexity from List; ScopeChain initially starts as a doubly-linked copy of List. (sourced)
- 2002-11-22 (2d8935af) replaced [[scope-chain-doubly-linked-copy]]: The doubly-linked ScopeChain required copying the full list when entering a function call scope (scope = func->scope().copy() then push activation); a singly-linked list with reference-counted shared tails eliminates all copy allocations because child scopes simply reference the parent tail, measured as 11% gain on iBench. (sourced)
- 2008-01-01 (3480a600) replaced [[scope-chain-all-heap-nodes]]: Every ScopeChain::push() allocated a heap ScopeChainNode even for transient pushes during function calls; embedding the top node directly in the ScopeChain object (m_initialTopNode) defers heap allocation to moveToHeap() called only on copy/assign, eliminating the allocation on the common single-push call path and yielding 1.019x SunSpider. (sourced)
- 2008-01-01 (dac291cc) replaced [[scope-chain-stack-top-optimization]]: The stack-allocated first-node optimization (m_initialTopNode + moveToHeap()) was reverted because it was causing correctness failures in practice ("breaking the world"), making heap-only ref-counted nodes the surviving form. (sourced)
- 2008-02-03 (13eb087e) replaced [[split-symbol-table-and-local-storage-save]]: The old design saved the symbol table (name->index hash map) and local storage (index->value vector) as separate operations, requiring two full copies of the hash table during back/forward cache creation; the new design iterates the symbol table once and emits {name,value,attributes} tuples into SavedProperties, eliminating the separate symbol table copy and the need for CachedPage to hold a SymbolTable. (sourced)
- 2008-10-01 (24a4e2f3) replaced [[arguments-object-via-jsactivation]]: Using 'arguments' in a function formerly forced creation of a JSActivation (full scope chain object); decoupling arguments storage into OptionalCalleeArguments call frame slot eliminates that cost and yields 19.1% on V8 Raytrace / 6.5% total V8 suite speedup. (sourced)
- 2010-09-16 (24dfdf84) replaced [[closure-feature-flag-activation]]: The ClosureFeature flag was set whenever any closure was present, forcing activation on functions that did not actually capture any variables from an enclosing scope; per-variable capture analysis lets functions skip activation when no variables from them are closed over. (sourced)
- 2010-09-23 (2c8a9583) replaced [[activation-copies-all-vars]]: The old code copied every local variable into the activation object regardless of capture status; once free-variable tracking was added to the parser, the bytecode generator can partition vars into captured (low indices) and non-captured (high indices) so the activation only needs to copy and mark m_numCapturedVars registers. (sourced)
- 2011-03-03 (fcac5b75) replaced [[variable-object-register-storage]]: Raw Register* in JSVariableObject symbol-table property storage bypassed the GC write barrier, allowing the GC to miss pointer updates when variables were written; replacing with WriteBarrier<Unknown>* makes all symbol-table writes go through the barrier. (sourced)
- 2012-08-26 (36e0a1ea) replaced [[jsactivation-malloc-backing-store]]: Switching JSActivation's register backing store from malloc (requiring a destructor) to GC CopiedSpace and its SharedSymbolTable from ref-counting to GC MarkedSpace eliminates the need for a destructor on activation objects, enabling GC to collect them without finalization overhead. (code)
- 2012-08-28 (3aa4cd77) replaced [[dispersed-variable-resolution]]: Multiple copies of variable-resolution logic (resolve, resolveSkip, resolveGlobal, resolveBase, resolveWithBase, resolveWithThis) existed independently in Interpreter.cpp, JITStubs.cpp, and CommonSlowPaths.h; introducing JSScope as a shared base class for all scope-chain objects centralised these into one authoritative implementation to eliminate duplication and enable a forthcoming scope-chain optimisation. (sourced)
- 2012-08-30 (3d62590a) replaced [[scope-chain-node]]: ScopeChainNode was a separate JSCell wrapper (holding next/object/globalObject/globalThis pointers) that was paired with every JSScope object, requiring two heap allocations per closure scope entry; merging its data into JSScope eliminates the wrapper object and one allocation per scope. (sourced)
- 2012-08-31 (82c4590f) replaced [[jsscope-inline-global-pointers]]: Storing m_globalData, m_globalObject, and m_globalThis in every JSScope instance duplicated three pointers across all activation objects; moving them to JSGlobalObject (one copy per global) and deriving them via structure()->globalObject() and MarkedBlock lookup reduced JSActivation from 128-byte to 64-byte size class, halving allocation cost. (sourced)
- 2013-05-09 (4e61ee97) replaced [[eval-code-cache-global-single]]: The single global CodeCache for eval was unsound: it could return a cached unlinked code block containing var-reference offsets computed under one function's scope for execution under a different function's scope (or inside |with|/|catch|), producing bogus variable lookups; replaced with per-UnlinkedCodeBlock caches (NonGlobalCodeCache, 20x smaller limits) used only when the eval is at the top of the scope chain with no intervening activation objects. (code)
- 2014-02-03 (82e9d20d) replaced [[per-call-bytecode-function-name-scope]]: We used to emit bytecode to push a name into local scope every time a function that needed such a name executed; now, we push the name into scope once on the function object, and leave it there. (sourced)
- 2014-08-20 (46386079) replaced [[implicit-activation-skip-scope-resolution]]: This is ground work for ensuring that all closed variable access is made through the function's activation. (sourced)
- 2015-03-26 (329af672) replaced [[captured-variables-can-reside-in-stack-frame]]: Declared variables now have an explicit, stable storage kind so a variable no longer moves between heap and stack during its lifetime. (sourced)
- 2016-03-15 (7e032b27) replaced [[lexical-environment-classinfo-identification]]: Distinct JSTypes for lexical, global lexical, and module environments replaced class-info dynamic casts so scope slow paths and cache setup can identify these scope kinds with cell type checks. (code)
- 2021-07-01 (49338f8f) replaced [[private-name-entry-use-declaration-bits]]: Private-name uses are tracked through the ordinary usedVariables mechanism so parser backtracking, source-provider-cache restoration, and runtime scope-chain lookup use the same rollback and capture propagation model as variables. (sourced)
- 2023-09-05 (14f1a47b) replaced [[global-declaration-instantiation-via-property-put]]: Global declaration instantiation needs separate CanDeclareGlobalX checks before any CreateGlobalXBinding side effect, because creating one binding before a later failed check is observable after catching the error. (sourced)
